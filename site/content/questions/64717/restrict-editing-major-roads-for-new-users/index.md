<p>As there will be a number of new people who will contribute to OSM for the first time. We can assume that there might be mistakes. So what is the recommended way to onboard new contributors?</p>

<div class="answer-body">
<p>There's no way to limit editing to certain objects. If you have the time, you could let your users log in to the "dev API" first, where they experiment and make changes without breaking anything. The "dev API" database is mostly empty; you could take the OSM data for a test area your mappers are familiar with and upload it to the dev API beforehand, then let them play with that, before you let them graduate towards the main API. The disadvantage is that the dev API doesn't have a rendered map, so it's all in the editor only.</p>
<p>Other than that, you will have to be a very good teacher to avoid major mishaps, and if you record the user names your mappers have created, you can go through their edits one by one after the event and fix what needs fixing (or revert what needs reverting).</p>
