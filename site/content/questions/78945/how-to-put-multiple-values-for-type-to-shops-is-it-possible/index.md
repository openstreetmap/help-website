<div class="answer-body">
<p>Yep, you just separate the values by a semi-colon: ";".</p>
<p>We have a company in the UK, <a href="https://en.wikipedia.org/wiki/Timpson_(retailer)">Timpson</a>, who started out as shoe repairers and also now major in key cutting as well, but also provide dry cleaning &amp; photo processing at some outlets. Usually, at a minimum, they are tagged shop=shoe_repair;key_cutting: but it has <a href="https://github.com/osmlab/name-suggestion-index/issues/3437">provoked plenty of discussion</a>. (In this case, as it is a major chain, adding <a href="https://wiki.openstreetmap.org/wiki/Key:brand">brand</a> tags can really help)</p>
<p>Other shops may be seen to have a main type of business with some sidelines: in these cases I'd stick with the main type of business.</p>

<p>Note that <a href="https://wiki.openstreetmap.org/wiki/Semi-colon_value_separator"></a><a href="https://wiki.openstreetmap.org/wiki/Semi-colon_value_separator"><code>https://wiki.openstreetmap.org/wiki/Semi-colon_value_separator</code></a> has, since its creation in 2010, recommended against semicolon-seperated values in "top-level" tags. It gives <strong><code>amenity=*</code></strong> as an example of a top-level tag, and I believe that <strong><code>shop=*</code></strong> is of the same tier.</p>
<p>My understanding is that the preference has generally been to choose a single value for the <strong><code>shop=</code></strong> key and encode other features and services with additional tags. Exactly which value gets top billing, and exactly which tags are used to encode the secondaries, would be matters of personal judgement and/or consensus.</p>
<p>Because this has been longstanding practice, if you do choose to encode multiple values in the <strong><code>shop</code></strong> tag, know that some software will not correctly parse these values.</p>

<div class="comment-text">
<p>The "Examples for established uses" section lists some keys where semicolon-separated values are widely used. The most common are opening hours for shops and amenities, cuisine for amenities, destination and turn lane tags for highways.</p>
<p>The main problem with using semicolons for shops is that a lot of existing software will only process exact matches of the shop tag values. Software <em>can</em> be written to search for substrings, but it can be a little tricky -- you don't want to find <strong><code>shop=carpet</code></strong> when you're searching for "car" or "pet." So the searches need to be slower and more complex to match a sub-value that may be at the beginning of the value string followed by a semicolon, or at the end preceded by a semicolon, or in the middle surrounded by semicolons. Many software developers have chosen to do without this extra complexity and simply search for exact matches.</p>
</div>

<p>BTW, you're correct that other mappers have used semicolon-separated values of the shop tag in the past, but it's infrequent. By far the most common is <strong><code>shop=convenience;gas</code></strong> which is in heavy use in France, but this would be much better mapped with two tags, <strong><code>shop=convenience</code></strong> and <strong><code>amenity=fuel</code></strong>.</p>
<p>But there are cases where I've found using a single value for shop inadequate. There's a hardware store near me that also sells vegetables and eggs. Other than using the <strong><code>description=*</code></strong> tag, there's no obvious way to encode that information in OSM. (Perhaps adding <strong><code>greengrocer=yes</code></strong> would be an option.)</p>
