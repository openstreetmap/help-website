<div class="comment-text">
<p>The thing in NoScript that interferes with JOSM remote control is the ABE, the Application Boundaries Enforcer. The ABE by default prevents pages on the internet from fiddling around with your local computer. You will find the ABE settings under the Advanced tab in the NoScript options dialog. Either you may disable the ABE completely (lazy, but somewhat bad idea) or write a customized ruleset that lets the commands to JOSM pass through.</p>
<p>The documentation is found here: <a href="http://noscript.net/abe/">http://noscript.net/abe/</a></p>

<div class="answer-body">
<p>also, if you're started having problems in April of 2014 or later, it could be the HTTPS certificate problem, as OpenStreetMap "Edit with remote control" started using HTTPS port on TCP/8112 (it used plaintext HTTP on TCP/8111 before, so there was no certificate and no problem).</p>
<p>To fix it, go to <a href="https://127.0.0.1:8112/">https://127.0.0.1:8112/</a> and accept invalid certificate.</p>
<p>See <a href="https://github.com/openstreetmap/openstreetmap-website/issues/740">github issue 740</a> for more info</p>

<p>If the problem is NoScript then you can to allow the request through the ABE blocker rules as follows:</p>
<p>1) Open the NoScripts options dialog,<br />
2) Goto the Advanced pane then the ABE sub-pane<br />
3) Change the SYSTEM rule to read:</p>
<pre><code>Site http://127.0.0.1:8111
Accept from *.openstreetmap.org
Deny
&#10;Site LOCAL
Accept from LOCAL
Deny</code></pre>

<div class="answer-body">
<p>If you are using Opera there are also security issues in the browser that you need to address. The warning it would normally show is hidden in an invisible iframe so the edit option silently fails. To overcome this you can go into Preferences (Ctrl-F12), Advanced, Security, Trusted Websites, Secure Internal Hosts and add 127.0.0.1 (<a href="http://openstreetmap.org">openstreetmap.org</a> edit option) and localhost (OSM Inspector). I guess there may be security issues that permanently flagging the sites as secure may raise, but as the links work with other browsers I don't think that it can be a major risk.</p>

<p>I had another problem with remote control not working today. Here is what I did to troubleshoot it:</p>
<p>First, I started JOSM, and verified it had "Enable remote control" and "Enable HTTPS support" enabled. It was, so I didn't need to enable it and restart JOSM. Next I did in shell:</p>
<pre><code>telnet localhost 8112</code></pre>
<p>if it said the following:</p>
<pre><code>Trying 127.0.0.1...
Connected to 127.0.0.1.
Escape character is &#39;^]&#39;.</code></pre>
<p>Then it would mean that JOSM, networking and local firewall on the machine are working, and that the problem is somewhere in the web browser (for which I'd then try to open "https://localhost:8112/" to debug further, whitelist certicates etc).</p>
<p>But in my case it hanged after "Trying 127.0.0.1..." and never showed "Connected to 127.0.0.1", which indicated that the problem is either JOSM or network/firewall. Then I did:</p>
<pre><code>netstat -tlpn</code></pre>
<p>to verify if JOSM is listening (that is on Debian GNU/Linux,on MS-windows it would be something like "netstat -a"?):</p>
<p>It should output something like:</p>
<pre><code>Proto Recv-Q Send-Q Local Address           Foreign Address         State       PID/Program name
tcp6       0      0 ::1:8111                :::*                    LISTEN      20297/java
tcp6       0      0 127.0.0.1:8111          :::*                    LISTEN      20297/java
tcp6       0      0 ::1:8112                :::*                    LISTEN      20297/java
tcp6       0      0 127.0.0.1:8112          :::*                    LISTEN      20297/java</code></pre>
<p>to indicate JOSM is listening on both port 8111 and 8112. But in my case the list was empty. If it wasn't empty, it would strongly indicate problem in local firewall, so I'd try disabling firewall and trying "telnet" again. But as it was empty, it indicated that problem is that JOSM didn't listen on that port - which is either problem in JOSM (less likely, as I was running stable version and lots of people would be making noise about it) or a problem in local machine networking.</p>
<p>I found a log in ~/.xsession-errors, and it contained:</p>
<pre><code>WARNING: Cannot start IPv4 remotecontrol server on port 8111: Cannot assign requested address (Bind failed)
WARNING: Cannot start IPv6 remotecontrol server on port 8111: Cannot assign requested address (Bind failed)
WARNING: Cannot start IPv4 remotecontrol https server on port 8112: Cannot assign requested address (Bind failed)
WARNING: Cannot start IPv6 remotecontrol https server on port 8112: Cannot assign requested address (Bind failed)</code></pre>
<p>then I started JOSM manually under strace with:</p>
<pre><code>strace -o /tmp/log.txt -s999 -tt -f java -jar josm-tested.jar
grep -w 8112 /tmp/log.txt</code></pre>
<p>and it showed:</p>
<pre><code>bind(38, {sa_family=AF_INET6, sin6_port=htons(8112), inet_pton(AF_INET6, &quot;::ffff:127.0.0.1&quot;, &amp;sin6_addr), sin6_flowinfo=htonl(0), sin6_scope_id=0}, 28) = -1 EADDRNOTAVAIL (Cannot assign requested address)</code></pre>
<p>"EADDRNOTAVAIL" means that no network interface on computer has requested address (127.0.0.1 in this case). I did "ip addr show" and found out that my computer really did not have "127.0.0.1" as the "lo" interface was down due to some other problem. So simple "/etc/init.d/networking start" fixed it (which is unlikely to be anybody elses problem, but hopefully those troubleshooting steps would help to narrow down problem).</p>
