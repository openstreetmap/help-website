<p>I am <em>thinking</em> of something like this: <code>memorial:relation:bicycle=yes</code>. Maybe a better word for "relation" should be used. Problem: the same tag for monuments makes no sense and using <code>monuments:relation:bicycle=yes</code> for them is not nice, too. One tag for memorials and monuments (and what else) would be good.</p>
<p>Maybe one tag similar to <code>commemorates:wikidata=*</code> (listed on <a href="https://wiki.openstreetmap.org/wiki/Tag:historic%3Dmemorial">https://wiki.openstreetmap.org/wiki/Tag:historic%3Dmemorial</a> ): <code>commemorates:category:bicycle=yes</code> or rather <code>commemorates:category:cycling=yes</code>. We have some of those <a href="https://taginfo.openstreetmap.org/search?q=commem">commem…</a> keys.</p>
<p>I guess binary tagging ("yes" and maybe "no") as values only could make sense here, as several categories could apply:</p>
<ul>
<li><code>commemorates:category:cycling=yes</code></li>
<li><code>commemorates:category:accident=yes</code></li>
<li><code>commemorates:category:death=yes</code></li>
</ul>
<p>But isn't that too much information already for the OSM database? I tend to think so. And which categories to use? Which depth level? I am not convinced of my own idea.</p>
