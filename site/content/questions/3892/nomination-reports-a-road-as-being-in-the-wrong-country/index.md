<div class="answer-body">
<p>This type of problem usually occurs if the information about where the way is situated (inside which administrative boundary) is not correct.</p>
<p>You can debug this by repeating the search on <a href="http://nominatim.openstreetmap.org/">http://nominatim.openstreetmap.org/</a> (Nominatim is the search engine behind the search on <a href="http://openstreetmap.org">openstreetmap.org</a>).</p>
<p>You get the same results, but here you can click on "details" to find out how Nominatim reached its result. Doing so, you get a map showing you the OSM object that Nominatim found for "Dieren", along with some details. There you can see that:</p>
<p>a) The city of Dieren exists as a real polygon in the OSM database, i.e. its exact boundaries are in the database. However, the boundaries of the city of Dieren in the OSM database do not include the Kolonieweg (they end just short of it). So its clear that searching for "Kolonieweg, Dieren" does not work.</p>
<p>b) For some reason, Nominatim seems to believe that Dieren and its surroundings are part of Bottrop (Germany). So from that point of view it's correct that you need to search for "Kolonieweg, Bottrop".</p>
<p>So to answer your question:</p>
<ul>
<li>The boundaries of Dieren in the OSM database need to be corrected (if the Kolonieweg really is inside Dieren).</li>
<li>For some reason Nominatim believes that Dieren is in Bottrop. Maybe someone else here can clarify why, otherwise a bug report against Nominatim should probably be filed.</li>
</ul>
<p><strong>Edit:</strong></p>
<p>I misunderstood the data in the database. There are two areas/polygons named "Dieren": one is a "place=village", and the other a "boundary = administrative". The "place=village" one does not include Kolonieweg, but the other one does, and that's enough for Nominatim. Thus there's no need to change anything.</p>
