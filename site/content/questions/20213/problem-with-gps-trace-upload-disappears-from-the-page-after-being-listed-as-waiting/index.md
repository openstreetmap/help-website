<div class="answer-body">
<p>I have had that happen too. If your GPS track does not have a timestamp for every lat/lon point, it will be rejected by the OSM server. I discovered that when I trisd to load a track that I had saved in Google Earth but not on my computer.</p>
<p>I converted my track back to GPX and tried to load it to OSM and had the very same result as you. Google Earth tosses the time stamps when you upload and save a GPX track. Many GPSrs, older Garmins like the 60 series in particular, also strip some of the data from a GPS track when you save it on the unit.</p>
<p>I did find a workaround if the track you want to use is important to get into OSM. Using GPSBabel in command line mode, you can force it to add dummy timestamps to a GPX file. If you're interested enough and are reasonably comfortable using a program from the Windows command window, read the GPSBabel help under Faketime. Here is the command line I used:</p>
<p>C:\Program Files (x86)\GPSBabel&gt;gpsbabel -i gpx -f c:\in.gpx -x track, faketime=f20080705130705+2 -o gpx -F c:\out.gpx</p>
<p>faketime format: YYYYMMDDHHMMSS+SS (offset each point)</p>
<p>Here I used Jul 05, 2008 at 13:07:05 for a start time with a 2 second offset added to each lat/lon pair.</p>
<p>GPSBabel is a sweet program. It can do many things besides convert GPS data from virtually any format into any other format.</p>

<p><span></span><span>@AlaskaDave</span>: "track that I had saved in Google Earth" - the google problem again. ;-) It is not very clear what the source of the track in Google Earth is - tracing their aerial imagery is not allowed for OSM. Can one also download data from Garmins with Google Earth?</p>
<p>Btw: it is not a bug that tracks without timestamps are rejected. The timestamps are useful to see which speed the gps track was recorded at! So, introducing a fake speed is <em>not</em> a good idea <em>unless it really "is important"</em> to have a track without speed info in OSM. If possible speed data should be uploaded.</p>
